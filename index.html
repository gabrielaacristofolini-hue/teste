QUIZZES HTML

01

O que é um QUIZ HTML?

No contexto de mídia programática, um quiz é um fluxo estratégico de captura de leads que guia o usuário por uma jornada interativa de perguntas ou etapas visuais, com o objetivo de capturar o lead e o redirecionar para sites monetizados.

Chamamos de Quiz HTML quando a construção desse fluxo ocorre via código HTML personalizado. Diferente de ferramentas prontas que limitam o criador a templates e configurações pré-definidas, o Quiz HTML oferece liberdade total: cada elemento visual, cada regra de fluxo e cada detalhe de performance podem ser ajustados sob medida para a campanha. As limitações nesse cenário, idealmente, são apenas as limitações que um código HTML pode ter.

Diferenciais Técnicos:

Personalização

Fluxo de perguntas, condições de resposta e regras de redirecionamento totalmente customizáveis.

Design

Layout adaptável a qualquer identidade visual e estratégia de campanha.

Performance

Código leve que garante carregamento rápido e reduz bounce rate.

Controle Total

Livre configuração de tags do Google, scripts de conversão e rastreamento em qualquer etapa do fluxo, dinâmicas de conversão, informações de payload e validação, entre outros.

02

Como funciona o Quiz HTML no molde SPUN?

Para além do design e do código visual, que são totalmente maleáveis, o código HTML precisa conter um bloco de configuração em JavaScript (o objeto CONFIG) que garanta seu funcionamento em harmonia com o ecossistema do MySPUN. O código atua como o "front-end" (a vitrine), enquanto a comunicação com o sistema ocorre através das seguintes diretrizes técnicas obrigatórias:

Vínculo com o Quiz Externo e o MySPUN

O quiz HTML opera conectado a um quiz externo criado previamente no sistema MySPUN. Para que os dados sejam registrados no quiz externo correto, o código precisa declarar o QUIZ_ID, o QUIZ_SLUG e o COMPANY_ID.

  • • QUIZ_ID: O número de identificação único gerado para esse quiz específico dentro do MySPUN.
  • • COMPANY_ID: O número de identificação da empresa no sistema MySPUN.
  • • QUIZ_SLUG: A slug específica configurada no quiz externo no MySPUN (ex: jn-us-emp-n01).

Endpoints de Validação e Registro

1
Endpoint de Validação de E-mail
https://quizzes.spun.com.br/api/validate/email

Uma rota padrão acionada em tempo real enquanto o usuário digita. Ela retorna as propriedades do e-mail (se é safe, blacklist, disposable ou duplicate).

2
Endpoint de Mirror (Registro)
https://quizzes.spun.com.br/api/lead/{slug}

Uma rota dinâmica baseada no slug do quiz externo. A URL completa desse endpoint pode ser copiada diretamente no MySPUN, no painel do quiz externo que está sendo usado. É para cá que o quiz envia, via requisição POST, o payload final, um pacote contendo o nome, e-mail, URL de origem, IDs de identificação e o objeto quizAnswers (que armazena todas as opções que o usuário clicou durante as perguntas). É nessa API que o lead é armazenado num primeiro momento.

Atenção Vale ressaltar que o quiz externo de onde esse código foi copiado precisa aceitar o domínio de origem do site; essa permissão de URL de origem é configurada diretamente no painel.

Rota de Redirecionamento (Destination URL)

O código precisa ter uma URL de destino configurada. Ao executar o redirecionamento final, o script captura automaticamente todos os parâmetros de rastreio (UTMs) presentes na URL original do anúncio e os anexa a essa URL de destino, garantindo a continuidade do tracking. É importante destacar aqui que geralmente usamos rotas que são também configuradas no MySPUN. Essas rotas de redirecionamento no sistema permitem inserir mais de uma URL final e definir qual a porcentagem de tráfego vai para cada.

Traqueamento e DataLayer (GTM)

A estrutura de rastreamento exige a instalação do script do Google Tag Manager na tag <head> da página. O código do quiz se comunica com o GTM via dataLayer.push. O sistema não dispara o pixel apenas no clique final; ele mapeia as respostas da API e envia eventos específicos de diagnóstico (como event: 'blacklist' ou event: 'duplicate') e o evento final de sucesso (event: 'conversaoValida'). Além disso, também devem ser enviados dados e eventos ao GTM em cada etapa do quiz, marcando o avanço nas perguntas, as respostas selecionadas e o engajamento do usuário com o fluxo.

Dinâmica de Conversão (Conversion on Registration)

O quiz possui uma variável de controle chamada CONVERSION_ON_REGISTRATION. Ela atua como um portão de segurança rigoroso para o disparo da conversão no GTM e define a regra exata de quando o pixel deve ser alimentado. Analisando o comportamento real do código, ela funciona da seguinte forma:

Se for true (Blindagem Total)

O evento de conversaoValida só será disparado se o lead cumprir o fluxo com 100% de perfeição. Isso significa que o usuário deve passar pelas validações locais, a API de e-mail deve responder sem instabilidades, o lead não pode cair em nenhum filtro de bloqueio (como blacklist ou e-mail descartável) e, obrigatoriamente, a requisição final para a API de Mirror deve retornar um sucesso (HTTP 200 OK) confirmando que o dado foi salvo no banco. Neste cenário (true), ocorrem muito mais redirecionamentos silenciosos (Soft Redirects). Se qualquer critério não for atingido ou se houver falha de rede no envio, o código ignora o disparo da conversão e apenas redireciona o usuário. Essa configuração protege o algoritmo da plataforma de anúncios, garantindo que ele seja otimizado exclusivamente com leads reais e perfeitamente registrados.

Se for false (Foco na Intenção)

A regra é afrouxada. O evento de conversão é disparado (e o redirecionamento ocorre) mesmo que a requisição final para o banco de dados falhe ou o registro seja ignorado. Nessa condição, o quiz entende que o usuário realizou o preenchimento, e a marcação no pixel prioriza a ação que ocorreu na página, independentemente de instabilidades no servidor de destino no momento do envio.

Soft Redirects (Redirecionamentos Silenciosos)

O quiz executa redirecionamentos sem disparar conversão em situações de anomalia. Por exemplo: se a API de validação de e-mail ficar fora do ar (gerando uma "aprovação forçada" por falha de comunicação) ou se a trava de Cooldown identificar que é um usuário enviando o quiz várias vezes seguidas, o script aborta a tentativa de registro. Ele redireciona o usuário para a rota final de destino imediatamente para não perder a chance de monetizar aquele usuário nas quedas, mas faz isso de forma silenciosa, blindando o pixel contra dados corrompidos ou spam.

Trava Anti-Spam (Cooldown)

O código possui uma configuração de COOLDOWN_MINUTES. Ele grava um registro temporal no localStorage do navegador do usuário. Se o mesmo usuário tentar preencher o quiz novamente dentro desse intervalo de tempo, o código bloqueia o envio na API e o envia silenciosamente para a oferta (soft redirect), evitando disparos duplicados no pixel e sobrecarga no banco de dados.

Aceitação de Leads Duplicados (Accept Duplicate)

Através do objeto de configuração, a chave booleana ACCEPT_DUPLICATE (true/false) permite gerenciar a entrada de leads que já existem no sistema. Ela define se o código deve aceitar, registrar e disparar a conversão para um lead que a API identificou como duplicado na base. Se marcado como false, esse usuário tem o registro bloqueado e cai na regra de redirecionamento silencioso.

Aceitação de E-mails Descartáveis (Accept Disposable)

Seguindo a mesma lógica de filtro, a chave booleana ACCEPT_DISPOSABLE (true/false) define se o sistema deve registrar e aceitar leads que utilizaram provedores de e-mail temporários ou descartáveis. Se configurado como false, o quiz ignora o registro na API e não dispara o pixel, redirecionando o usuário de forma silenciosa para não perder o potencial do clique.

Script Específico de Conversão

Para que a plataforma de anúncios reconheça o sucesso do fluxo e otimize a campanha corretamente, o registro do pixel é feito através de um script de disparo específico enviado ao Google Tag Manager (GTM). No momento do sucesso da captura do lead, o código deve acionar e executar obrigatoriamente o seguinte bloco:

JavaScript dataLayer.push({
    'event': 'conversaoValida',
    'eventCallback': performRedirect,
    'eventTimeout': 2000
});

É de extrema importância garantir que esse script de conversão esteja perfeitamente inserido no código e seja chamado na lógica exata de sucesso. A chave event com o valor 'conversaoValida' é o gatilho preciso que o painel do GTM espera receber para contabilizar a ação. Além disso, as chaves eventCallback e eventTimeout são fundamentais, pois garantem que o usuário seja redirecionado em segurança para a rota final da oferta logo após o disparo efetivo do pixel. Toda a inteligência de otimização das campanhas depende exclusivamente da execução intacta desse script.

03

Configuração de Quiz Externo no MySPUN

Como vimos na parte anterior, o quiz HTML precisa se conectar a um quiz externo configurado dentro do MySPUN. É esse quiz externo que atua como o receptor oficial dos dados capturados. Para garantir o funcionamento perfeito, existem regras claras de criação e preenchimento.

Reutilização de Quizzes Externos

Antes de criar um novo quiz externo, é preciso observar a estrutura da operação. Trabalhamos com apenas um quiz externo por suboperação. Se a suboperação em questão já possui um quiz externo ativo no sistema, ele deve obrigatoriamente ser reaproveitado. A criação de um novo quiz externo é exclusiva para lançamentos de novas suboperações. Ao criar um quiz externo, é estritamente necessário configurar os pontos de identificação, segurança e rotas de distribuição. Esses campos garantem que o sistema reconheça a origem do tráfego, proteja os dados e entregue o lead no destino final corretamente.

Identificação e Nomenclatura

Para manter a organização do ecossistema, os campos de identificação exigem uma padronização exata baseada na estrutura: operação, país, nicho e suboperação.

Campo Padrão Exemplo
Título do Quiz Letras maiúsculas com espaços JN US CC N01
Slug URL Letras minúsculas com hifens jn-us-cc-n01

Permissões

Domínios: Configuração essencial, onde precisa configurar a URL exata onde o seu HTML ficará hospedado. O quiz externo bloqueia automaticamente qualquer tentativa de envio de dados que venha de um domínio diferente dos listados aqui.

Destinos de API e Distribuição de Leads

Após o recebimento do dado, o MySPUN precisa saber para onde encaminhar esse lead final. A configuração exata dessas APIs é o que garante que o setor de retenção receba os leads corretamente.

  • • Destinos de API Final: O campo principal onde entram as APIs. É obrigatório configurar a divisão do tráfego para que a soma desses destinos feche sempre em exatamente 100%.
  • • Destino Secundário de API Final: Uma rota de segurança auxiliar. É muito como um plano de contingência para salvar o lead caso o servidor do destino principal falhe.

Ao finalizar a criação do quiz externo, configurando corretamente o título, a slug, os domínios e as APIs, o painel estará pronto para uso. Imediatamente após a criação, o sistema disponibiliza as informações essenciais para a sua integração. Será possível identificar o número gerado (o ID do quiz externo) para preencher a variável QUIZ_ID nas configurações do código, bem como copiar a URL completa do endpoint para inserir na rota de mirror do seu código HTML. Com esses passos concluídos, a comunicação entre a página e o MySPUN estará perfeitamente estabelecida.

04

Configuração de Rotas no MySPUN

As rotas são links de redirecionamento que nos permitem enviar os usuários capturados na página para outros destinos. Essa funcionalidade é fundamental no nosso ecossistema, pois possibilita dividir o tráfego de forma percentual entre mais de uma queda.

Reutilização de Rotas

Seguindo a mesma lógica aplicada aos quizzes externos, as rotas são separadas por suboperação. Caso a suboperação já possua uma rota ativa no sistema, ela deve obrigatoriamente ser reaproveitada. A criação de uma nova rota ocorre exclusivamente para novas suboperações.

Padronização de Domínio

É muito importante prestar atenção ao domínio escolhido para a criação do link. Esse detalhe precisa ser conferido especificamente para cada operação antes de iniciar a configuração da rota em si.

Identificação e Nomenclatura

Para as rotas, o padrão de nomenclatura possui um elemento extra, pensado para facilitar a identificação rápida dentro do painel do MySPUN.

Campo Padrão Exemplo
Tag Op / País / Nicho / AQUISICAO / Subop (MAIÚSCULAS) JN US CC AQUISICAO N01
Slug URL Minúsculas com hifens jn-us-cc-aquisicao-n01

Configuração de Destinos

  • • URLs de Destino: Campo dedicado à inserção dos links finais para os quais a rota vai redirecionar o usuário.
  • • Percentual de Redirecionamento: Configuração obrigatória que define o volume exato de tráfego que cada URL cadastrada receberá.

Integração Final: Com a configuração salva, a rota estará pronta para uso. O último passo é copiar a URL final gerada pelo sistema e colá-la na configuração de URL de destino (geralmente identificada como Destination URL) diretamente no código HTML do seu quiz. É esse passo final que garante que o fluxo leve o usuário para a queda correta após a captura do lead.

06

Criação de Layout e Design do Quiz

Para além das configurações técnicas (que devem seguir a padronização descrita anteriormente), é possível personalizar totalmente o layout e o design do quiz. Atualmente, não é necessário programar manualmente os códigos HTML. Em vez disso, utilizamos Inteligência Artificial (IA) para construir e gerar o código da página como um todo.

1
Base de Criação

Ao planejar um layout novo, é recomendado partir do código de um quiz ativo e bem configurado. Entregar esse material como base permite que a IA absorva a estrutura e entenda perfeitamente os padrões operacionais que devem ser mantidos.

2
Engenharia de Prompts

Nesse processo, é fundamental trabalhar com comandos muito bem desenvolvidos. Isso garante que a IA altere apenas os elementos visuais da página, mantendo estritamente imutáveis os blocos de validação, o objeto de configuração, o traqueamento e as requisições. É essa trava que garante que o novo design continue totalmente funcional.

3
Revisão de Código

Mesmo utilizando excelentes prompts, sempre existe a probabilidade de ocorrerem erros ou interpretações equivocadas por parte da IA. Portanto, é obrigatório analisar com calma o resultado gerado para identificar, entender e corrigir eventuais quebras no fluxo.

4
Aprovação Final

Com o código do novo quiz finalizado, ele deve obrigatoriamente ser enviado ao responsável pela manutenção de quizzes da operação. Essa etapa final garante uma auditoria técnica apurada, atestando se todos os parâmetros estão corretos e se não há pontos destoantes antes de a página ir para o ar.

07

Integração e Configuração Final (Setup do Código)

Após o processo de revisão e aprovação pelo responsável, o próximo passo é garantir que as informações estruturais do quiz, previamente criadas e validadas no painel do MySPUN, sejam inseridas com precisão no código do quiz.

Todas essas chaves essenciais devem ser atualizadas diretamente no objeto principal de configuração, chamado CONFIG, e no cabeçalho do arquivo HTML:

Variável CONFIG Ação
QUIZ_ID Inserir o número de identificação único.
QUIZ_SLUG Inserir a slug criada no sistema.
COMPANY_ID Atribuir o número de identificação da empresa.
ENDPOINT_EMAIL_VALIDATION Inserir a rota padrão (ex: https://quizzes.spun.com.br/api/validate/email).
ENDPOINT_QUIZ_MIRROR Colar a URL do quiz externo copiada do painel.
DESTINATION_URL Colar o link final gerado nas configurações de rota.

Google Tag Manager (GTM)

Para além das configurações do MySPUN, é estritamente necessário garantir que a estrutura de rastreamento esteja apontando para o local certo. Você deve localizar o script do GTM instalado na tag <head> da página e substituir o código base de identificação padrão (ex: GTM-K3QRK9BF) pelo ID oficial correspondente à operação atual. Isso garante que os eventos enviados via dataLayer alimentem o pixel correto da campanha.

08

Hospedagem e Publicação do Quiz

Com o código configurado e validado, o passo final é a publicação da página no domínio correto. Este processo exige atenção total à padronização das URLs e às permissões de segurança do ecossistema.

Seleção do Domínio e Subdomínio

Cada operação possui um subdomínio específico que deve ser obrigatoriamente utilizado. É fundamental garantir que este domínio esteja devidamente configurado nas permissões do Quiz Externo dentro do MySPUN.

Criação da Página no WordPress

Após a escolha do domínio, acesse o painel administrativo do WordPress para criar a nova página. O título da página deve seguir o padrão de nomenclatura que será utilizado também na slug do quiz. Atualmente, todos os modelos seguem uma estrutura específica. A nomenclatura funciona da seguinte forma:

Padrão de Nomenclatura A estrutura exata é funil-[nome]-[modelo]. A parte do nome precisa reforçar uma característica visual ou funcional do quiz, seguida da identificação do modelo.

Exemplos: funil-slider-m1 ou funil-multistep-m1.

Configuração com Elementor

  • • Template da Página: Ao editar com o Elementor, acesse as configurações da página e defina o layout como Elementor Canvas. Isso garante uma página totalmente limpa, sem cabeçalhos ou rodapés do tema original.
  • • Inserção do Código: Insira um componente de Bloco HTML na página e cole o código completo que foi revisado e configurado anteriormente.

Publicação e Testes Finais

Após publicar a página, verifique novamente se a slug da URL está seguindo o padrão estabelecido. Com a página no ar, é obrigatório realizar uma revisão prática de todas as validações de campo e das solicitações de API. Certifique-se de que o lead está sendo registrado corretamente e que os redirecionamentos estão operando antes de liberar o quiz para uso oficial nas campanhas.

09

Revisões e Testes Finais

Com a página oficialmente publicada no domínio, é obrigatório realizar uma bateria de testes rigorosos. Essa etapa garante a saúde da captação e o funcionamento perfeito do ecossistema. Para isso, você deve utilizar as ferramentas de desenvolvedor do navegador (modo Inspecionar) e o modo Debug do Google Tag Manager (GTM).

Testes via Modo Inspecionar (Network e Console)

Através da aba Network do navegador, você deve simular preenchimentos e monitorar o comportamento técnico do código na prática:

  • • API de Validação de E-mail: Confirme se as requisições estão sendo feitas em tempo real e se o sistema está retornando os status corretos (safe, blacklist, disposable, duplicate).
  • • Endpoint de Mirror (Registro): Valide se a requisição POST final está enviando o pacote de informações completo e se o payload contém todos os dados esperados (respostas do quiz, IDs de configuração e URL de origem).
  • • Mapeamento de Cenários: Simule erros propositais para testar em quais cenários o soft redirect (redirecionamento silencioso) está ocorrendo de fato, garantindo que o código blinde o pixel contra leads corrompidos.

Testes via GTM Debug

Com o painel de depuração do GTM aberto em paralelo, é necessário auditar o comportamento do pixel:

  • • Disparo de Conversão: Confirme se o evento principal de conversaoValida está sendo disparado normalmente e apenas nos cenários adequados, respeitando as regras estabelecidas nas variáveis de configuração.
  • • Eventos de Diagnóstico: Verifique se as tags auxiliares de bloqueio ou anomalias também estão sendo lidas pelo dataLayer.

Repasse de UTMs

Depois disso, ao cair na URL de destino final após o preenchimento, você deve verificar a barra de endereços do navegador para confirmar o repasse de UTMs. É obrigatório garantir que todos os parâmetros de rastreio originais da campanha tenham sido transferidos e anexados corretamente ao link da oferta.

10

Catalogação e Registro Interno

Após a publicação e a validação técnica bem-sucedida, a etapa final obrigatória consiste na catalogação do quiz na Planilha de Controle Interno. Este procedimento assegura a organização dos ativos e o histórico de atualizações da operação.

Registro na Planilha de Quizzes

Acesse a planilha pelo link oficial e preencha os campos referentes ao novo quiz produzido:

  • • Identificação do Ativo: Insira o nome do modelo, utilizando a slug oficial.
  • • Arquivo e Versão: Registre o quiz HTML correspondente e indique a sua versão atual, seguindo a sequência correta (v1, v2, v3 etc).

Documentação por Operação

É fundamental que o registro também seja realizado pelas operações que utilizarão o quiz. Todas as informações de configuração, como IDs e chaves específicas de setup, devem ser catalogadas detalhadamente. Essa documentação centralizada permite uma visão clara de quais ativos estão rodando em cada frente e facilita futuras manutenções ou auditorias técnicas.
